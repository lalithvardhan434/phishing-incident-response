## Field-by-Field Forensic Analysis

### Return-Path
```
Return-Path: <victoria1119995@gmail.com>
```
**Analysis:** Return-Path matches the From address. In real phishing
campaigns, attackers often set Return-Path to a different address to
intercept bounce messages and avoid detection. Match here is expected
since this is a self-contained lab simulation.

---

### Received (Hop Trace)
```
Received: from Boddalas-MacBook-Air.local
          ([2405:201:c01b:f0f8:c8f3:50bd:7e01:386e])
        by smtp.gmail.com with ESMTPSA
```
**Analysis:** The email originated from `Boddalas-MacBook-Air.local`
— the GoPhish server (SIEM_SERVER/MacBook M2). The originating IPv6
address `2405:201:c01b:f0f8:c8f3:50bd:7e01:386e` is the Mac's public
IPv6 address assigned by the home router.

**Forensic Value:** This single Received header traces the email
directly back to the sending machine. In a real investigation, this
IPv6 address would be submitted to your ISP or threat intel platform
to identify the attacker's infrastructure.

**Key Finding:** Only ONE Received hop — email went directly from
GoPhish on the Mac to Gmail's SMTP server. No intermediate relays.
This is consistent with GoPhish sending directly via authenticated SMTP.

---

### TLS Encryption
```
version=TLS1_3 cipher=TLS_AES_128_GCM_SHA256 bits=128/128
```
**Analysis:** Email was transmitted from GoPhish to Gmail over
TLS 1.3 — the strongest available transport encryption. This means
the email content was encrypted in transit between the Mac and Gmail.

**Important Distinction:** TLS encrypts the SMTP transport layer
only. The phishing landing page used plain HTTP — credentials were
transmitted unencrypted from the victim's browser to GoPhish.

---

### X-Mailer
```
X-Mailer: gophish
```
**Analysis:** This is the most obvious indicator of a phishing
simulation tool. GoPhish inserts its own X-Mailer header by default.

**Real-World Note:** In actual red team engagements, this header
would be customised or removed to avoid detection. Leaving it as
`gophish` would immediately flag the email in any security-aware
environment. This is a configuration oversight worth noting in the
incident report.

**Detection Opportunity:** A Splunk alert on `X-Mailer: gophish`
in email logs would instantly identify GoPhish campaigns.

---

### Message-ID
```
Message-Id: <1781592649200822000.51307.541218723310399823
             @Boddalas-MacBook-Air.local>
```
**Analysis:** Message-ID is generated by GoPhish using the local
hostname (`Boddalas-MacBook-Air.local`). Legitimate corporate email
systems generate Message-IDs using the mail server's FQDN.

**Red Flag:** `@Boddalas-MacBook-Air.local` in the Message-ID
reveals that the email was generated by a local machine, not a
corporate mail server. This is a strong phishing indicator that
an email gateway or analyst should catch.

---

### Date / Timezone
```
Date: Tue, 16 Jun 2026 12:20:50 +0530
```
**Analysis:** Email dated 16 Jun 2026 at 12:20 IST (+0530).
GoPhish server timezone is India Standard Time (IST), consistent
with the lab location in Dubai/India region.

**Forensic Value:** Timezone in headers can help attribute attack
origin geography. IST (+0530) indicates India or nearby region.

---

### Subject Analysis
```
Subject: [Action Required] Reset Your Password Immediately
```
**Phishing Indicators Identified:**
| Indicator | Technique |
|---|---|
| `[Action Required]` | Urgency — bypasses rational thinking |
| `Reset Your Password` | Fear — implies account is compromised |
| `Immediately` | Time pressure — prevents victim verification |
| Square brackets | Visual attention grabbing |

**MITRE Mapping:** Social engineering via urgency aligns with
T1566 — Phishing. Attackers exploit psychological pressure to
reduce victim's critical evaluation of the email.

---

### Phishing Link Analysis
```
href="http://192.168.29.191:80?rid=z6QPCo3"
```
**Analysis of each component:**
| Component | Value | Significance |
|---|---|---|
| Protocol | http:// | No encryption — credentials sent in plaintext |
| Host | 192.168.29.191 | Raw IP address — not a domain name |
| Port | 80 | Default HTTP — no TLS |
| Parameter | rid=z6QPCo3 | GoPhish recipient tracking ID |

**Red Flags:**
- Legitimate password reset links use HTTPS and a domain name, never a raw IP
- `rid=` parameter is GoPhish's tracking mechanism — identifies which
  recipient clicked, enabling open/click tracking per victim

---

### Tracking Pixel
```
<img alt='' style='display: none'
     src='http://192.168.29.191:80/track?rid=z6QPCo3'/>
```
**Analysis:** GoPhish embeds a hidden 1x1 pixel image. When the
email is opened, the victim's email client loads this image,
sending an HTTP GET request to the GoPhish server. This is how
GoPhish recorded "Email Opened" at 09:20:08 UTC — two minutes
after delivery.

**Forensic Value:** The tracking pixel timestamp (09:20:08) vs
first link click (09:25:53) shows the victim spent approximately
5 minutes reading the email before clicking — consistent with
someone carefully reading a suspicious email.

## Analyst Conclusion

The email headers contain multiple forensic artifacts that conclusively
identify this as a phishing email originating from a GoPhish server
running on a local MacBook (Boddalas-MacBook-Air.local).

Most significant finding: The `X-Mailer: gophish` header is an
immediate detection signal. A production email gateway with header
inspection rules would flag and quarantine this email before delivery.

The tracking pixel mechanism explains the 2-minute gap between
delivery and "Email Opened" — a real-world technique used by
both phishing operators and legitimate email marketers.

The 5-minute gap between opening and first click suggests the
victim read the email carefully — yet still clicked. This reinforces
the effectiveness of urgency-based social engineering even against
cautious users, and highlights why technical controls (MFA, URL
filtering) are more reliable than user vigilance alone.
